# RULES — i18n Module

**Scope:** Locale (language) management, database-backed locale CRUD, settings page, locale selector UI.
**Spec:** [specs/internationalization.md](../specs/internationalization.md)

---

## 1. Entity Rules

### 1.1 Locale Entity
```go
type Locale struct {
    ID        uint      // auto-increment PK (gorm_id)
    Code      string    // BCP 47 language tag (e.g., "en", "vi", "zh-cn")
    Name      string    // human-readable label (e.g., "English", "Tiếng Việt")
    IsDefault bool      // exactly one locale must be true
    CreatedAt time.Time
    UpdatedAt time.Time
}
```

### 1.2 Code Rules
- Format: lowercase, 2-5 characters, letters and hyphens only
- Regex: `^[a-z]{2,5}(-[a-z]{2,5})?$`
- Unique constraint
- **Immutable after creation** — renaming would orphan all documents referencing the old code
- Used as the relate key: `Document.Locale` stores this value

### 1.3 Name Rules
- Required, 1-100 characters
- Human-readable, displayed in dropdowns and settings
- Can be updated freely (unlike code)

### 1.4 Default Locale
- **Exactly one** locale must have `IsDefault = true` at all times
- Setting a new default automatically clears the previous (usecase layer, not DB trigger)
- Unsetting the only default → rejected (409)
- If no default set (edge case during migration) → fall back to first alphabetically

---

## 2. Domain Invariants

### 2.1 Minimum Locale Count
- At least one locale must exist at all times
- **NEVER** allow the last remaining locale to be deleted
- Deleting the last locale → 409 Conflict

### 2.2 Deletion Guard
- A locale cannot be deleted if documents reference it
- Check `DocumentRepository.CountByLocale(ctx, code)` before deletion
- If documents exist → 409 Conflict with count in message
- If deleting the default locale → next locale (alphabetically) becomes new default

### 2.3 Database is Source of Truth
- `SUPPORTED_LOCALES` env var is **removed** after migration
- Locales are fully managed via the admin UI
- **NEVER** hardcode locale lists in frontend — always fetch from API

---

## 3. Migration Rules

### 3.1 First Startup Migration
- If locales table/collection is empty:
  - If `SUPPORTED_LOCALES` env var present → seed from it (first = default)
  - If env var absent → seed with `{ code: "en", name: "English", isDefault: true }`
- After seeding, env var is ignored
- Migration runs once, idempotently

## 4. Repository Rules

### 4.1 LocaleRepository
```go
Create(ctx, locale) error
FindByCode(ctx, code) (*Locale, error)
FindAll(ctx) ([]*Locale, error)
FindDefault(ctx) (*Locale, error)
Update(ctx, locale) error
Delete(ctx, code) error
ClearDefault(ctx) error
```

### 4.2 Key Methods
- `FindDefault` → returns locale with `isDefault == true`; `ErrNotFound` if none
- `ClearDefault` → sets `is_default = false` for all locales (called before setting new default)
- Both MongoDB and GORM implementations required

### 4.3 Extended DocumentRepository
- `CountByLocale(ctx, locale string) (int64, error)` — for deletion guard

## 5. API Contract Rules

### 5.1 Routes
| Method | Route | Auth | Description |
|---|---|---|---|
| `GET` | `/api/locales` | Public | List all locales (Locale[] objects) |
| `POST` | `/api/locales` | `super_admin` | Create locale |
| `PUT` | `/api/locales/:code` | `super_admin` | Update locale (name, isDefault) |
| `DELETE` | `/api/locales/:code` | `super_admin` | Delete locale |

### 5.2 Breaking Change
- `GET /api/locales` previously returned `string[]` → now returns `Locale[]`
- Frontend `useLocales` hook returns `Locale[]` (not `string[]`)

### 5.3 Response Shapes
- List: `[ { "code", "name", "isDefault", "createdAt", "updatedAt" } ]`
- Create: `201 Created` → Locale object
- Update: `200 OK` → Locale object
- Delete: `204 No Content`

### 5.4 Error Responses
- Duplicate code → 409 Conflict
- Invalid code format → 422
- Delete with documents → 409 with count
- Unset only default → 409
- Non-super_admin on write → 403

### 5.5 Update Rules
- `code` cannot be changed (path param only, not in body)
- `isDefault: true` → clears previous default
- `isDefault: false` rejected if this is the only default

## 6. Frontend Rules

### 6.1 InternationalizePage
- Route: `/admin/settings/internationalize`
- Access: `super_admin` only (ProtectedRoute)
- Sidebar: listed under Settings group
- Table: Code, Name, Default (star), Actions (Edit, Delete)
- Add: Dialog with Code (text), Name (text), Default (checkbox) — Code enabled
- Edit: Same dialog, Code disabled (read-only)
- Delete: Confirmation dialog; error toast if documents exist
- Last locale's Delete button disabled

### 6.2 LocaleSelector Component
```typescript
interface LocaleSelectorProps {
  value: string
  onChange: (code: string) => void
}
```
- Uses Shadcn `Select` component
- Shows language **name** in options (not code)
- Pre-selects default locale when `value` is empty
- Compact styling, fits in toolbar

### 6.3 CollectionListPage Integration
- Locale dropdown in toolbar between title and action buttons
- Always visible (even with one locale)
- Changing locale: updates `activeLocale` → resets pagination → refetches documents
- Default locale pre-selected on mount
- Selected locale passed to all document operations

### 6.4 State Management
- Query keys: `['locales']` for list
- Invalidate `['locales']` after any mutation
- `useLocales` returns `Locale[]` objects
- Mutation hooks: `useCreateLocale`, `useUpdateLocale`, `useDeleteLocale`
- `useDeleteLocale` shows error toast on 409

## 7. Testing Rules (i18n-Specific)

### 7.1 Backend Usecase Tests
- CRUD happy paths
- Default-swap logic (set new default clears old)
- Deletion guards (last locale, locale with documents)
- Code validation (format, uniqueness)
- Seed-from-env-var logic
- Code immutability after creation

### 7.2 Backend Handler Tests
- 401 for unauthenticated
- 403 for non-super_admin on write ops
- 422 for invalid code format
- 409 for duplicate code, delete with documents, unset only default
- GET returns Locale objects (not strings)

### 7.3 Backend Repository Tests (GORM)
- `ClearDefault` + `Create` with `IsDefault`
- `FindDefault` when no default set
- Unique constraint on code

### 7.4 Frontend Tests
- InternationalizePage: renders table, add/edit/delete dialogs, default indicator
- LocaleSelector: renders names, calls onChange with code
- CollectionListPage: locale dropdown visible, changing resets pagination

## 8. Boundaries Summary

| Rule | Detail |
|---|---|
| **Always** | `code` is the relate key — what `Document.Locale` stores |
| **Always** | Display language `name` in all user-facing dropdowns and tables |
| **Always** | Validate locale code format at usecase layer (not just frontend) |
| **Always** | Invalidate `['locales']` cache after any mutation |
| **Always** | At least one locale must exist at all times |
| **Always** | Exactly one locale must be the default |
| **Always** | Check `CountByLocale` before locale deletion |
| **Always** | Fetch locales from API — never hardcode |
| **Never** | Allow editing a locale's code after creation |
| **Never** | Allow zero locales in the system |
| **Never** | Store locale name in documents — only the code |
| **Never** | Hardcode locale lists in frontend |
| **Never** | Use `SUPPORTED_LOCALES` env var after migration |
| **Ask first** | Before deleting a locale referenced by documents |
| **Ask first** | Before changing the default locale |
